# What it is

OKF is an open format for representing **knowledge** — the metadata, context, and curated
insight surrounding data and systems. Google Cloud published it on 2026-06-13; the current
version is **v0.1 Draft**.

From the spec itself:

> "The format is intentionally minimal: a directory of markdown files with YAML frontmatter.
> There is no schema registry, no central authority, and no required tooling. If you can `cat`
> a file, you can read OKF; if you can `git clone` a repo, you can ship it."

No runtime, no SDK, no central registry. If you can read a file, you can read OKF.

# Why it exists

The announcement names the problem as knowledge being scattered — metadata catalogs (each with
its own API), wikis, shared drives, code comments and docstrings, and "the heads of a few senior
engineers".

> "Every agent builder is solving the same context-assembly problem from scratch, every catalog
> vendor is reinventing the same data models, and the knowledge itself is locked behind whichever
> surface created it."

# Four design principles (spec §1)

- **Readable** by humans without tooling
- **Parseable** by agents without bespoke SDKs
- **Diffable** in version control
- **Portable** across tools, organizations, and time

# Core structure

- **A concept's ID is its path in the bundle minus `.md`** (`tables/users.md` → `tables/users`)
- **`type` is the only required frontmatter field.** Recommended: `title`, `description`,
  `resource`, `tags`, `timestamp` (ISO 8601)
- **Type values are not centrally registered.** Spec examples: `BigQuery Table`, `API Endpoint`,
  `Metric`, `Playbook`, `Reference`. Defining a fixed taxonomy is an explicit non-goal
- Reserved filenames: `index.md` (directory listing), `log.md` (reverse-chronological change
  history). Both optional
- Links should be absolute from the bundle root (`/tables/customers.md`) — stable across moves
- Version is declared as `okf_version: "0.1"` in the root `index.md` frontmatter — the only
  place frontmatter is allowed in an `index.md`

# Conformance is deliberately permissive — this matters

A bundle conforms if just three things hold: (1) every non-reserved `.md` has parseable YAML
frontmatter, (2) every frontmatter has a non-empty `type`, (3) reserved files follow their
structure.

And the spec explicitly forbids consumers from rejecting a bundle over: missing optional fields,
unknown `type`, unknown extra keys, **broken links**, or a missing `index.md`.

> "Consumers MUST tolerate broken links — a link whose target does not exist in the bundle is
> not malformed; it may simply represent not-yet-written knowledge."

This permissiveness is intentional: bundles must stay useful while they grow, get refactored,
and are partially generated by agents.

# Relationship to other formats

Spec §10 says it directly: OKF is intentionally close to LLM "wiki" repositories, personal
knowledge tools like Obsidian and Notion, and "metadata as code" approaches. The difference is
that OKF is **specified** — it pins down the minimum rules needed for interoperability without
dictating tooling.

OKF does not replace domain schemas like Avro, Protobuf, or OpenAPI. It *references* them.
